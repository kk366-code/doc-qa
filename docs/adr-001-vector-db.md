# ADR-001: ベクトルDBに pgvector を選択した理由

**ステータス**: 採用  
**作成**: 2026-05-09  
**作成者**: KT

---

## コンテキスト

エンタープライズ向けの社内ドキュメント Q&A システムを構築する。
ベクトル検索が必要な RAG パイプラインを実装するにあたり、
専用ベクトルデータベース（Pinecone, Qdrant, Weaviate）か、
既存の PostgreSQL 拡張（pgvector）かを選定する必要があった。

本システムは既存の PostgreSQL 環境への統合を前提として設計する。

---

## 評価した選択肢

| 選択肢 | 特徴 |
|--------|------|
| **pgvector** | PostgreSQL 拡張。ACID 特性を保ちながらベクトル検索が可能 |
| Pinecone | マネージド専用ベクトルDB。高スループット・高次元に強い |
| Qdrant | OSS 専用ベクトルDB。高い ANN 精度。Rust 製で高速 |
| Weaviate | OSS + マネージド。GraphQL API でスキーマ管理が強力 |

---

## 決定

**pgvector を採用する。**

---

## 理由

### 1. 既存インフラとの統合コスト
PostgreSQL を既に本番運用していることを前提とすれば、
pgvector を追加するだけで済み、新たなデータストアの運用・監視コストが発生しない。

### 2. ACID 特性を活かせる
ドキュメントのメタデータ（ファイル名・アップロード日時・アクセス権限）と
ベクトルを同一トランザクションで管理できる。
専用ベクトルDBでは 2 フェーズコミットが必要になるか、
結果整合性を許容する設計変更が必要になる。

### 3. SQL との親和性
既存のデータと JOIN でき、フィルタリング（「特定部門のドキュメントのみ検索」等）を
SQL 一本で書ける。Pinecone では metadata filter API が別途必要。

### 4. HNSW インデックス（pgvector 0.5+）
バージョン 0.5 以降は HNSW をサポートしており、
中小規模（〜100 万ベクトル）では Qdrant に迫る ANN 精度が出る。

---

## トレードオフ

| 観点 | pgvector | 専用ベクトルDB |
|------|----------|----------------|
| 最大スケール | 〜1000 万ベクトル程度 | 数十億ベクトル |
| ANN 精度 | HNSW で十分（中小規模） | Qdrant はより高精度 |
| 運用コスト | PostgreSQL の追加コスト最小 | 別途インフラが必要 |
| フィルタリング | SQL で自由自在 | 専用 API が必要 |
| マネージドサービス | AWS RDS / Aurora 対応 | Pinecone は完全マネージド |

---

## 再評価トリガー

以下の条件を満たした時点で専用ベクトルDBへの移行を検討する：

- ベクトル数が **1000 万件を超えた場合**
- 検索レイテンシが **p99 で 500ms を超えた場合**
- フィルタリングの複雑さが SQL の限界に達した場合
